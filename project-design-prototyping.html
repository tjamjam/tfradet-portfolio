The Challenge

Static mockups couldn't keep pace

I brought a decade of design experience and a well-honed Figma workflow. High-fidelity mockups, detailed component specs, design system documentation—the standard toolkit of a senior product designer.

The product was a complex enterprise platform. Data tables with thousands of rows, multi-step workflows, real-time dashboards. Every design review became a game of "imagine if..." rather than "look at this."

Questions kept surfacing that static mockups couldn't answer: How does this table behave with 10,000 rows? What happens when a user filters, then groups, then searches? Can we see what the loading states actually feel like?

Static mockups couldn't show behavior

The handoff gap was costly

  • • Features shipped that didn't match the envisioned experience
  • • Iteration cycles stretched as we discovered interaction issues late
  • • Stakeholder alignment was difficult without experiencing actual behavior
  • • Engineering invested in building things we'd realize weren't right

Figma showed what something looked like, but not how it felt to use. Not how it performed with real data. Not the micro-interactions that made experiences feel polished.

The fastest path to design alignment was working prototypes, not higher-fidelity mockups.

The Evolution

From Figma-first to vibe coding

Phase 1: Vibe Coding Emerges

I started experimenting with "vibe coding"—using AI assistants to rapidly translate design intent into functional code.

The approach was simple: describe what I wanted to an AI, iterate on the output conversationally, and end up with working code in minutes instead of hours. Not production code—prototype code.

Phase 2: Building the Bridge

Vibe coding in isolation wasn't enough. I needed to connect it to our design system, make it reproducible, and enable collaboration.

The key insight was to export Figma tokens and use them to theme component libraries. Prototypes felt like our product, not generic UI.

Phase 3: Team Capability

Rather than mandating change, I built the infrastructure first, demonstrated value through my own prototypes, and made collaboration feel accessible. The design team now iterates on functional prototypes together, reviewing PRs and pushing design forward in code.

Principles

What guided the journey

Prototypes Over Perfection

A working prototype that's 80% right teaches more than a perfect mockup that's 100% wrong about behavior.

Design System as Bridge

By exporting Figma tokens and applying them to code components, the prototype looked like the design. Visual fidelity came from the system, not manual effort.

Collaboration Through Code

GitHub became a design collaboration space. PRs became design reviews. Comments on code became design feedback.

Reflections

What I learned

On Tool Evolution

The tools we use shape what we can imagine. Moving from static to functional prototyping didn't just speed up execution—it changed what I could conceive. I started thinking in interactions, not screens.

On AI as Design Partner

AI didn't replace design thinking—it accelerated it. I still made all the design decisions. The AI handled the translation to code, freeing me to focus on what mattered: does this interaction serve the user?

On Leading Through Building

I could have written documentation about why designers should learn to code. Instead, I built the system that made it possible and demonstrated the value. The team followed because they could see it working.

On Bridging Worlds

The magic was in the bridge—Figma tokens flowing into themed components. Neither world was abandoned; they were connected. Design stays in Figma for exploration. Prototypes prove behavior in code. Each tool does what it does best.

"The question isn't 'Figma or code?' It's 'which tool serves this moment best?'—and now I have both."
